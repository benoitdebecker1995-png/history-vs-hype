Conduct a token usage audit using all available methods. The goal is to identify which commands and agents consume the MOST tokens so routing decisions target the biggest savings.

**Step 1: Install and run ccusage** (primary audit tool)
```bash
npm install -g ccusage
```
Run ccusage against Claude Code usage logs. On Windows, logs are typically at:
- `%APPDATA%\claude-code\` or `%USERPROFILE%\.claude\` or `%USERPROFILE%\.claude-code\`
- Search for JSONL log files if standard paths don't work

If ccusage fails to find logs or is incompatible, fall back to manual analysis.

**Step 2: Manual profiling via session analysis**
Since ccusage may not have data for all commands (depends on log format), supplement with:
- Read through .claude/commands/*.md files -- estimate token footprint based on:
  - Length of the command prompt itself (input tokens from the .md file)
  - Typical number of file reads the command triggers (each file = input tokens)
  - Typical output length (output tokens)
  - Whether command uses Task tool (spawns subagent = multiplied tokens)
- Read through .claude/agents/*.md files -- estimate token footprint based on:
  - Agent system prompt length
  - Typical number of tool calls
  - Output complexity

**Step 3: Estimate relative consumption**
Create a ranked table of all 19 tasks (13 commands + 6 agents) with:
- Estimated input tokens per invocation (prompt + context files read)
- Estimated output tokens per invocation
- Frequency of use (daily, weekly, per-video, rarely)
- Total estimated monthly token consumption
- Percentage of total

Use realistic estimates. For example:
- /script reads STYLE-GUIDE.md (~3000 tokens), VERIFIED-RESEARCH.md (varies, 5000-20000 tokens), generates 3000-8000 word script output (4000-10000 tokens)
- /status reads STATE.md (~300 tokens), generates ~200 token response
- /analyze reads video data + generates analysis report (~2000 input, ~1500 output)

**Step 4: Pareto analysis**
Identify the top 3-5 commands that consume ~80% of total tokens. These are the optimization targets.

Using the TOKEN-AUDIT.md results and Phase 13.1 model assignments, create a routing classification document that maps every command/agent to a routing tier.

**Step 1: Define routing tiers**

| Tier | Description | Target |
|------|-------------|--------|
| **Claude-Only** | Quality-critical, complex reasoning, creative synthesis | Keep on Claude (Opus/Sonnet) |
| **Routable-Free** | Simple/mechanical, quality validated or low-risk | OpenRouter free models (Gemini 2.0 Flash, Llama 3.3 70B) |
| **Routable-Local** | NOT FEASIBLE on this hardware | Ollama (blocked: 14.9GB RAM, no GPU) |

**Step 2: Classify every task**

For each of the 19 tasks (13 commands + 6 agents):
1. Start with Phase 13.1 tier (Haiku/Sonnet/Opus)
2. Apply routing eligibility:
   - Opus tasks -> Claude-Only (quality-critical, no routing)
   - Sonnet tasks -> Claude-Only UNLESS output is structured/mechanical (then Routable-Free with validation)
   - Haiku tasks -> Routable-Free candidates (mechanical work, free models can handle)
3. Consider task-specific factors:
   - Does the task use Task tool? (may not work with non-Claude)
   - Does the task need access to codebase files? (may need Claude Code integration)
   - Does the task produce user-facing creative output? (quality risk)
   - Is the task time-sensitive? (latency concerns with OpenRouter free tier)

**Step 3: Create routing recommendation per task**

For each Routable-Free task, specify:
- Recommended free model (e.g., "google/gemini-2.0-flash-exp:free" or "meta-llama/llama-3.3-70b-instruct:free")
- Quality risk level (LOW, MEDIUM, HIGH)
- Validation needed before routing (yes/no, what to test)
- Estimated token savings if routed

For each Claude-Only task, explain WHY it must stay on Claude (one sentence).

**Step 4: Calculate potential savings**

Sum up estimated monthly tokens from Routable-Free tasks. Compare to total. Express as percentage savings and approximate dollar amount (using Anthropic pricing: ~$3/M input, ~$15/M output for Sonnet; ~$0.25/M input, ~$1.25/M output for Haiku).

**Step 5: Routing Strategy Decision**

This section is the FINAL ROUTING STRATEGY DECISION for the phase -- not a recommendation, but the chosen approach with justification.

Write a "## Routing Strategy Decision" section that contains:
1. **Chosen tool:** State the selected routing tool (e.g., "Claude Code Router with OpenRouter" or "Wave Terminal" or "Native Claude /model switching"). This is a DECISION, not a recommendation.
2. **Justification:** Why this tool was chosen over alternatives (2-3 sentences referencing hardware constraints, daily usage patterns, and complexity tradeoff).
3. **Configuration path:** Exact steps to set up the chosen tool (install command, config file location, environment variables needed).
4. **Rejected alternatives:** List other tools considered and one-line reason each was rejected.

Given hardware constraints (no local models feasible), the decision will likely be:
- **Primary:** Claude Code Router with OpenRouter as secondary provider
- **Alternative:** Wave Terminal for manual side-by-side testing
- **NOT recommended:** Native Ollama (insufficient RAM for useful models)

Note the OpenRouter free tier limitation (50 req/day default, 1000/day with $10 credits). Calculate whether daily routed command frequency fits within limits.
