**Researched:** 2026-02-07
**Domain:** AI model routing and token optimization
**Confidence:** MEDIUM

## Summary

Multi-model token optimization for Claude Code involves routing lightweight tasks to free or local models while reserving Claude for complex work. Three routing approaches exist in 2026: Claude Code Router (proxy-based routing with 8+ providers), native Ollama integration via Anthropic API (added v0.14.0), and Wave Terminal (side-by-side AI configurations). Token auditing is available through built-in `/context` command, Admin API endpoints, and community tools like ccusage.

The optimization opportunity is significant: developers achieve 40-70% cost reduction through strategic routing. Local models like Qwen 2.5 Coder (32B) score 70.6% on SWE-Bench compared to Claude Sonnet 4.5's 77.2%, making them viable for routine coding tasks. OpenRouter offers 18 free models (rate-limited to 50-1000 requests/day depending on credit purchases) as an alternative to local hosting.

**Primary recommendation:** Start with token audit using `/context` and ccusage to identify high-consumption commands, then evaluate Claude Code Router for maximum flexibility before committing to Wave Terminal or native Ollama integration.

## Architecture Patterns

### Pattern 1: Audit-First Optimization

**What:** Measure before optimizing. Audit token consumption to identify which commands/agents consume most tokens.

**When to use:** Always. This is the mandatory first step before any routing decisions.

**Process:**

1. **Built-in audit:**
   ```bash
   # In Claude Code session
   /context  # Shows per-action token breakdown
   /cost     # Shows current usage estimate
   ```

2. **Historical audit:**
   ```bash
   # Using ccusage CLI
   ccusage analyze ~/.claude-code/
   ccusage report --by-command --sort tokens
   ```

3. **Organization-level audit (requires Admin API key):**
   ```bash
   curl https://api.anthropic.com/v1/organizations/usage_report/messages \
     -H "x-api-key: sk-ant-admin-..." \
     -d '{"start_date": "2026-01-01", "end_date": "2026-02-07"}'
   ```

**Metrics to track:**
- Input tokens (uncached)
- Cached input tokens
- Cache creation tokens
- Output tokens
- Tokens per command type
- Tokens per agent invocation

**Expected outcome:** Identify top 3-5 commands consuming 80% of tokens (Pareto principle).

### Pattern 2: Task Classification by Complexity

**What:** Classify each command/agent into complexity tiers to determine routing eligibility.

**Complexity tiers:**

| Tier | Model Target | Task Characteristics | Examples |
|------|-------------|---------------------|----------|
| **Simple** | Haiku or free models | Pattern matching, templating, file operations | `/status`, `/help`, `/fix` (SRT correction) |
| **Standard** | Sonnet or Qwen 2.5 Coder | Analysis, structured output, moderate reasoning | `/analyze`, `/verify`, `/patterns` |
| **Complex** | Opus (keep on Claude) | Creative synthesis, deep reasoning, quality-critical | `/script`, specialized agents |

**Phase 13.1 baseline (current assignments):**


**Routing candidates for Phase 28.1:**

- **Haiku tasks → Qwen 2.5 Coder 7B or OpenRouter free tier:** High likelihood of success (simple operations)
- **Sonnet tasks → Qwen 2.5 Coder 32B:** Medium likelihood (needs quality validation)
- **Opus tasks → Keep on Claude:** Quality-critical, worth the premium cost

### Pattern 4: Quality Validation Protocol

**What:** Before routing tasks to non-Claude models, validate output quality systematically.

**Validation approach:**

1. **Baseline capture (Week 1):**
   - Run command with Claude (current state)
   - Save output as gold standard
   - Document quality expectations

2. **Alternative model test (Week 2):**
   - Run same command with candidate model (Qwen, OpenRouter free)
   - Compare outputs side-by-side
   - Score quality: PASS / ACCEPTABLE / FAIL

3. **Criteria for routing approval:**
   - PASS: Output matches or exceeds Claude quality → Route immediately
   - ACCEPTABLE: Minor quality loss but acceptable → Route with monitoring
   - FAIL: Significant quality degradation → Keep on Claude

4. **Monitoring post-routing:**
   - User spot-checks routed commands weekly
   - Escalate to Claude if output quality degrades
   - Document which model types work for which task types

**Example quality rubric for `/analyze` command:**

| Criterion | Weight | Claude Baseline | Qwen 32B Test | Score |
|-----------|--------|----------------|---------------|-------|
| Data accuracy | 40% | 100% match with API | 100% match | PASS |
| Insight quality | 30% | 3 actionable insights | 2 actionable insights | ACCEPTABLE |
| Markdown formatting | 20% | Perfect | Perfect | PASS |
| Execution time | 10% | 8 sec | 12 sec | ACCEPTABLE |
| **Overall** | - | - | - | **ACCEPTABLE** |

Conclusion: Route `/analyze` to Qwen 32B with weekly spot-checks.

### Anti-Patterns to Avoid

- **Routing without auditing:** Don't guess which commands are expensive—measure first.
- **Quality blindness:** Don't assume free models match Claude quality—validate systematically.
- **All-or-nothing migration:** Don't route everything at once—phase gradually by complexity tier.
- **Ignoring latency:** Don't route time-sensitive commands through proxies—measure response times.
- **Over-engineering:** Don't build complex routing logic for marginal savings—focus on 80/20.

## Common Pitfalls

### Pitfall 1: Routing Quality-Critical Tasks Too Aggressively

**What goes wrong:** User routes `/script` command (Opus-tier creative work) to Qwen 32B to save tokens. Output quality drops, scripts require significant manual revision, net time loss.

**Why it happens:** Token cost looks high on paper, seems like optimization opportunity. Benchmark scores (Qwen 70.6% vs Claude 77.2% on SWE-Bench) look "close enough."

**How to avoid:**
- Maintain "sacred cow" list of quality-critical commands (script writing, fact-checking)
- Calculate time value: "Would I spend 30 min fixing output to save $0.50 in tokens?"
- Use Pattern 4 validation before routing any creative/quality-sensitive task

**Warning signs:**
- User manually edits outputs more frequently after routing
- Commands require re-runs to get acceptable quality
- "Close enough" becomes "needs significant work"

### Pitfall 2: Free Tier Rate Limit Surprise

**What goes wrong:** User routes 6 Haiku commands to OpenRouter free tier. Hits 50 req/day limit by noon. All routed commands fail, workflow blocked.

**Why it happens:** OpenRouter free tier has two rate limits: 50 req/day (default) or 1000 req/day (if purchased $10+ credits). User didn't account for command frequency.

**How to avoid:**
- Audit command frequency before routing: `ccusage report --by-command --timeframe today`
- Calculate: (commands per day) × (routed commands) = daily request count
- If > 50/day: Either purchase $10 OpenRouter credits or use local Ollama
- Set up monitoring: Track 429 rate limit errors

**Warning signs:**
- Commands fail intermittently during workday
- Error logs show "rate limit exceeded" from OpenRouter
- Workflow productive in morning, blocked in afternoon

### Pitfall 3: Local Model Hardware Requirements Ignored

**What goes wrong:** User installs Ollama, pulls Qwen 2.5 Coder 32B (20GB quantized), system runs out of RAM, Ollama crashes, commands timeout.

**Why it happens:** Model sizes are deceptive. "32B parameters" sounds abstract. Actual RAM requirement: 20-24GB for Q4 quantization, 40GB+ for full precision.

**How to avoid:**

**Hardware requirements by model size:**

| Model | Quantization | RAM Required | GPU VRAM | Use Case |
|-------|--------------|--------------|----------|----------|
| Qwen 2.5 Coder 7B | Q4_K_M | 4-6 GB | Optional | Simple tasks, lightweight machines |
| Qwen 2.5 Coder 32B | Q4_K_M | 20-24 GB | Optional (faster with) | Standard tasks, modern workstations |
| Qwen 2.5 Coder 72B | Q4_K_M | 40-48 GB | 24GB+ VRAM recommended | Complex tasks, high-end systems |

**Pre-installation checks:**
```bash
# Check available RAM
free -h  # Linux
vm_stat  # macOS
systeminfo | findstr Memory  # Windows

# Check GPU VRAM (if using GPU acceleration)
nvidia-smi  # NVIDIA GPUs
```

**Right-sizing:**
- 16GB RAM system → Use 7B models only
- 32GB RAM system → Can run 32B models comfortably
- 64GB+ RAM system → Can run 72B models

**Warning signs:**
- System swap usage spikes when running Ollama
- Commands take 5+ minutes to respond
- Ollama process killed by OOM (Out of Memory) errors

### Pitfall 4: Proxy Latency Compounds

**What goes wrong:** User routes through Claude Code Router → OpenRouter → Free model. Each hop adds 200-500ms latency. Simple commands that took 2 sec now take 6 sec. Workflow feels sluggish.

**Why it happens:** Proxy architecture adds network hops. OpenRouter free tier often queues requests during peak times. Latency perception is nonlinear (6 sec feels 3x worse than 2 sec).

**How to avoid:**
- Measure baseline latency before routing: `time claude-code-command`
- Test routed latency: `time claude-code-command` (with router active)
- Set acceptable latency threshold: e.g., "commands must complete in <5 sec"
- If proxy latency unacceptable: Switch to native Ollama integration (zero proxy overhead)

**Latency comparison (typical):**

| Routing Path | Network Hops | Typical Latency | Acceptable For |
|--------------|--------------|-----------------|----------------|
| Direct to Claude | 1 (API) | 1-3 sec | All commands |
| Claude Code Router → Ollama local | 2 (proxy, local) | 2-4 sec | Most commands |
| Native Ollama | 1 (local) | 1-2 sec | All commands |
| Claude Code Router → OpenRouter free | 3 (proxy, OpenRouter, provider) | 4-8 sec | Non-time-sensitive only |

**Warning signs:**
- Commands feel noticeably slower after routing
- User hesitates to run commands due to wait time
- Response time variability (fast sometimes, slow others)

### Pitfall 5: Configuration Drift and Debugging Hell

**What goes wrong:** User enables Claude Code Router with 3 providers. Months later, API key expires, config file syntax breaks, hard to remember which commands route where. Debugging takes hours.

**Why it happens:** Configuration complexity grows over time. Multiple config files (Claude Code Router, Ollama, Wave Terminal). No version control. No documentation.

**How to avoid:**

**Configuration management strategy:**

1. **Version control all configs:**
   ```bash
   git init ~/.claude-code-router
   git add config.json
   git commit -m "Initial routing config"
   ```

2. **Document routing decisions:**
   ```markdown
   # ROUTING-STRATEGY.md

   ## Current Configuration (as of 2026-02-07)

   ### Commands Routed to Qwen 32B Local:
   - /analyze (validated 2026-02-01, quality: ACCEPTABLE)
   - /patterns (validated 2026-02-03, quality: PASS)

   ### Commands Kept on Claude:
   - /script (quality-critical, Opus required)
   - /verify (fact-checking accuracy critical)

   ### Validation Results:
   [Link to validation spreadsheet]
   ```

3. **Health check script:**
   ```bash
   # check-routing-health.sh
   echo "Checking Claude API..."
   curl -s https://api.anthropic.com/v1/messages \
     -H "x-api-key: $ANTHROPIC_API_KEY" \
     -d '{"model":"claude-haiku-4-5","messages":[{"role":"user","content":"test"}],"max_tokens":10}' \
     | jq '.type'

   echo "Checking Ollama..."
   curl -s http://localhost:11434/api/tags | jq '.models[].name'

   echo "Checking OpenRouter..."
   curl -s https://openrouter.ai/api/v1/auth/key \
     -H "Authorization: Bearer $OPENROUTER_API_KEY" \
     | jq '.data.label'
   ```

4. **Backup configs before changes:**
   ```bash
   cp ~/.claude-code-router/config.json ~/.claude-code-router/config.json.backup-$(date +%F)
   ```

**Warning signs:**
- Commands fail with cryptic error messages
- Can't remember which commands are routed where
- API key rotation breaks workflow
- Config file syntax errors after manual edits

### Example 4: Quality Validation Script

```python
# validate-routing-quality.py
import subprocess
import json
from difflib import SequenceMatcher

def run_command_with_model(command, model_config):
    """Run Claude Code command with specified model configuration."""
    env = os.environ.copy()
    if model_config['type'] == 'ollama':
        env['ANTHROPIC_BASE_URL'] = 'http://localhost:11434/v1'
    elif model_config['type'] == 'router':
        env['CLAUDE_CODE_ROUTER_MODEL'] = model_config['router_model']

    result = subprocess.run(
        ['claude-code', command],
        capture_output=True,
        text=True,
        env=env,
        timeout=60
    )
    return result.stdout

def compare_outputs(baseline, test):
    """Compare two command outputs for similarity."""
    similarity = SequenceMatcher(None, baseline, test).ratio()
    return {
        'similarity': similarity * 100,
        'verdict': 'PASS' if similarity > 0.95 else 'ACCEPTABLE' if similarity > 0.85 else 'FAIL'
    }

# Validation configuration
commands_to_validate = [
    '/analyze VIDEO_ID',
    '/patterns',
    '/verify SCRIPT.md'
]

models_to_test = [
    {'name': 'Claude Sonnet (baseline)', 'type': 'cloud'},
    {'name': 'Qwen 32B Local', 'type': 'ollama', 'model': 'qwen2.5-coder:32b'},
    {'name': 'Gemini 2.0 Flash Free', 'type': 'router', 'router_model': 'openrouter,google/gemini-2.0-flash-exp:free'}
]

# Run validation
results = []
for command in commands_to_validate:
    print(f"Testing: {command}")
    baseline_output = run_command_with_model(command, models_to_test[0])

    for model in models_to_test[1:]:
        test_output = run_command_with_model(command, model)
        comparison = compare_outputs(baseline_output, test_output)
        results.append({
            'command': command,
            'model': model['name'],
            'similarity': comparison['similarity'],
            'verdict': comparison['verdict']
        })

# Generate report
with open('validation-report.json', 'w') as f:
    json.dump(results, f, indent=2)

print("\nValidation Report:")
for result in results:
    print(f"{result['command']} → {result['model']}: {result['verdict']} ({result['similarity']:.1f}% similar)")
```
