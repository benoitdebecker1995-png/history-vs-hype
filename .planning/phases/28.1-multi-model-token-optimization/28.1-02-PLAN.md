  <name>Task 1: Install and Configure Routing Tool</name>
  <files></files>
  <action>
Install Claude Code Router and configure it with Anthropic (primary) and OpenRouter (secondary provider for free models).

**Step 1: Install Claude Code Router**
```bash
npm install -g @datartech/claude-code-router
claude-code-router init
```

If npm install fails (package name changed, deprecated, etc.), fall back to:
- Check if Claude Code has built-in model switching (the `/model` command)
- Check if Wave Terminal is already installed and can be configured
- Document what worked and what didn't

**Step 2: Configure providers**

Create configuration with two providers:
1. **Anthropic** (primary) -- uses existing ANTHROPIC_API_KEY
2. **OpenRouter** (secondary) -- requires OPENROUTER_API_KEY

Configuration should map:
- Default: Anthropic Claude Sonnet (current behavior)
- Think/complex: Anthropic Claude Opus
- Background/simple: OpenRouter free model

Use the research examples as configuration reference (Pattern 3, Path A in 28.1-RESEARCH.md).

**Step 3: Handle OpenRouter API key**

The user needs an OpenRouter account and API key. Two scenarios:
- If OPENROUTER_API_KEY is already set in environment, use it
- If not, document what the user needs to do (create account at openrouter.ai, get API key)

Do NOT block on the API key. If it's not available:
1. Configure the router with Anthropic-only for now
2. Leave OpenRouter config as a placeholder with clear instructions
3. The checkpoint in Task 2 will verify the user has set it up

**Step 4: Test basic router functionality**

Verify the router starts and can route a simple request:
```bash
claude-code-router start
# or whatever the startup command is
```

Test with a trivial prompt to confirm routing works.

**Step 5: Document the setup**

Record in VALIDATION-RESULTS.md (setup section):
- What was installed (version numbers)
- Configuration file location
- How to start/stop the router
- Any issues encountered and workarounds
- Environment variables needed

**Windows-specific considerations:**
- Use forward slashes or escaped backslashes in config paths
- Check if claude-code-router works in MSYS_NT/Git Bash environment
- If Node.js path issues, use full path to npx/node

<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 2: Validate Routing Quality</name>
  <what-built>
Claude has installed a routing tool and will now test 2-3 commands from the "Routable-Free" tier (identified in Plan 01) by running them through both Claude and a free model, comparing outputs side-by-side.

Before validation, Claude will:
1. Pick 2-3 commands from the Routable-Free tier in ROUTING-CLASSIFICATION.md
2. Run each command with Claude (baseline) and capture output
3. Run each command with the free model and capture output
4. Compare outputs and score: PASS / ACCEPTABLE / FAIL
5. Write results to VALIDATION-RESULTS.md
6. Update ROUTING-CLASSIFICATION.md with validation status using this explicit logic:
   - For each tested task: if verdict = **FAIL**, change its routing tier to "Claude-Only" with reason: "Quality validation failed [date]"
   - For each tested task: if verdict = **PASS** or **ACCEPTABLE**, add `validation_status: VALIDATED` with date and confirmed free model target
   - For untested Routable-Free tasks: leave routing tier unchanged, add `validation_status: UNTESTED` (can be validated in future)
7. Calculate final savings estimate based on validated routing decisions only

The validation results will include:
- Side-by-side output comparison for each tested command
- Quality scores per criteria (accuracy, formatting, completeness)
- Final verdict per command (safe to route / keep on Claude)
- Updated savings estimate reflecting only VALIDATED routing decisions
  </what-built>
  <how-to-verify>
1. Review VALIDATION-RESULTS.md at `.planning/phases/28.1-multi-model-token-optimization/VALIDATION-RESULTS.md`
2. Check that each tested command has Claude baseline output and free model output shown
3. Verify the quality assessments make sense to you (do the free model outputs look acceptable?)
4. Review the updated ROUTING-CLASSIFICATION.md -- do the routing decisions feel right?
5. Check the savings estimate -- is it meaningful enough to justify maintaining the routing setup?

**If ALL tested commands score FAIL:** This is a valid outcome. It means the current command set is not suitable for free-model routing on this hardware/provider configuration. The phase still succeeds -- the finding is documented, ROUTING-CLASSIFICATION.md is updated to mark all tested tasks as Claude-Only, and the recommendation becomes: keep all tasks on Claude until free model quality improves. No routing infrastructure needs to be maintained.

If the routing tool installation failed or OpenRouter is not configured yet:
- Claude will document what happened and what manual steps remain
- You can decide whether to proceed with the routing setup or defer

Key question: Is the token savings worth the complexity of maintaining a routing setup?
