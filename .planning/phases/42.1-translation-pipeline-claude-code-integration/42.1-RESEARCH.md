## Summary

The translation pipeline (`tools/translation/`) currently contains 5 modules that make direct Anthropic SDK calls: `translator.py`, `cross_checker.py`, `legal_annotator.py`, `surprise_detector.py`, and `verification.py`. Each module initializes an `anthropic.Anthropic(api_key=...)` client, loads credentials via `env_loader.py`, and calls `client.messages.create()` directly. This requires Pro API users to maintain a separate `ANTHROPIC_API_KEY` in `.env` — the only pain point this refactor solves.

The project already has a well-established integration pattern: Python tools handle pure data processing, and Claude Code (via slash commands) handles all LLM calls natively. The `research.md` command at line 270 makes this explicit: "Claude reads this command, then runs the Python tool via Bash. Do NOT attempt to import Python directly from the command file — run it as a subprocess or Bash execution." The NLM ingestion tool (`nlm_ingest.py`) is the most recent example — it does zero LLM work; Claude Code itself does the parsing and writing.

The architecture decision is therefore clear: **extract all prompts and LLM calls out of the Python modules into the slash command layer**. Python modules become pure data processors (text parsing, structure detection, file I/O, formatting). Claude Code does all LLM inference using its native `create_message` context. This is not a new pattern — it is the pattern already used by every other tool in the workspace.

**Primary recommendation:** Move all prompts and `client.messages.create()` calls from Python into slash command instructions. Python modules become stateless data processors that receive pre-built prompts and return structured data. The `/translate` slash command (new) orchestrates the full pipeline.

### Pattern 1: Prompt Extraction Pattern (The Core Refactor)

**What:** Python module builds the structured data payload (text, context, clause IDs). Claude Code receives the payload, constructs the prompt inline (or from the slash command template), calls its native LLM, and parses the response. Python only validates and formats the result.

**When to use:** Every module that currently does `self.client.messages.create(...)`

**Current pattern (to be removed):**
```python
# In translator.py — BEFORE refactor
class Translator:
    def __init__(self, model='claude-sonnet-4-20250514'):
        key_result = load_api_key()
        self.client = anthropic.Anthropic(api_key=key_result['key'])

    def translate_clause(self, clause_text, full_document, source_language, clause_id, document_context=None):
        response = self.client.messages.create(
            model=self.model,
            max_tokens=2000,
            temperature=0.3,
            system=system_prompt,
            messages=[{"role": "user", "content": user_prompt}]
        )
        return self._parse_translation_response(response.content[0].text)
```

**New pattern (after refactor):**
```python
# In translator.py — AFTER refactor
class TranslationDataBuilder:
    """Pure data processor — no LLM calls, no API keys."""

    def build_translation_payload(self, clause_text, full_document,
                                  source_language, clause_id,
                                  document_context=None):
        """Returns structured dict for Claude Code to use."""
        highlighted_doc = self._highlight_clause(full_document, clause_text)
        user_prompt = self._build_translation_prompt(
            highlighted_doc, source_language, clause_id, document_context
        )
        return {
            'clause_id': clause_id,
            'original': clause_text,
            'system_prompt': TRANSLATION_SYSTEM_PROMPT,
            'user_prompt': user_prompt
        }

    def parse_response(self, response_text, clause_id, original_text):
        """Parse Claude's response text into structured result."""
        translation, notes = self._parse_translation_response(response_text)
        return {
            'clause_id': clause_id,
            'original': original_text,
            'translation': translation,
            'notes': notes
        }
```

The slash command then:
1. Calls `build_translation_payload()` to get the prompt
2. Calls Claude natively with the prompt (no Python API call)
3. Calls `parse_response()` with the text result
4. Proceeds to next step

### Pattern 2: Halt-and-Ask on Failure

**What:** When a pipeline step fails (e.g., cross-check finds significant discrepancy), stop and report to user. Do not auto-continue. User decides whether to accept, revise, or abort.

**When to use:** Cross-check step when `severity == 'significant'`; any step that returns an error dict.

**Implementation in slash command:**
```
## On Cross-Check Discrepancy
If cross_check result contains severity='significant':
  - Display the discrepancy clearly
  - Ask: "Cross-check found a significant discrepancy in [clause_id].
    - Claude translation: [text]
    - DeepL translation: [text]
    - Difference: [explanation]
    Options:
    1. Accept Claude's translation and continue
    2. Use DeepL's version
    3. Review manually (I'll pause here)
    What would you like to do?"
  - Wait for user response before continuing
```

### Pattern 3: Incremental Rollout (One Module First)

**What:** Refactor `translator.py` first. Validate end-to-end with a real document. Then apply the same pattern to `cross_checker.py`, `legal_annotator.py`, `surprise_detector.py`, and `verification.py` in sequence.

**Why:** Translator is the most critical module (used by every downstream step). If the prompt extraction pattern works there, it works everywhere.

**Order:**
1. `translator.py` → proof of concept
2. `cross_checker.py` → second (semantic comparison LLM call removal)
3. `legal_annotator.py` → third
4. `surprise_detector.py` → fourth
5. `verification.py` → fifth (scholarly comparison calls)
6. Delete `env_loader.py` and clean up `smoke_test.py`

### Pattern 4: Model Assignment per Step

Based on task complexity, assign models per step:

| Step | Module | Recommended Model | Reasoning |
|------|--------|-------------------|-----------|
| Translation | translator.py | `claude-sonnet-4-6` | Nuanced legal/historical language requires quality; sonnet is the right tradeoff |
| Cross-check semantic comparison | cross_checker.py | `claude-haiku-4-5` (if available) or `claude-sonnet-4-6` | JSON comparison task, simpler reasoning |
| Legal annotation | legal_annotator.py | `claude-sonnet-4-6` | Requires legal knowledge, terminology expertise |
| Surprise detection | surprise_detector.py | `claude-sonnet-4-6` | Requires contextual reasoning about narratives |
| Verification scholarly comparison | verification.py | `claude-sonnet-4-6` | Document analysis task |

**Note:** Use latest aliases only. Current pinned dates in code (`claude-sonnet-4-20250514`) must be replaced with `claude-sonnet-4-6` or equivalent current alias. Verify current alias names before implementing.

### Anti-Patterns to Avoid

- **Dual code paths:** Do NOT keep the `anthropic.Anthropic()` client as a fallback. Clean break only. The user constraint is explicit.
- **Python subprocess calling `claude` CLI:** The slash command IS Claude Code — it runs natively, no subprocess needed.
- **Keeping env_loader.py for other purposes:** The user decision is to delete it entirely. Check dependents first (see below).
- **Auto-continuing on pipeline failure:** The user explicitly wants halt-and-ask behavior. Never silently continue past errors.
- **Batch mode:** User wants one document at a time. Do not add parallelization.

## Common Pitfalls

### Pitfall 1: Prompt Ownership Confusion

**What goes wrong:** Prompts remain embedded in Python modules (e.g., `TRANSLATION_SYSTEM_PROMPT = "..."` in `translator.py`) but the LLM call moves to the slash command. The Python module now has no way to "run" these prompts, creating dead code and confusion about where prompts should live.

**Why it happens:** The easiest refactor removes only the `client.messages.create()` call and leaves the prompt constants in Python.

**How to avoid:** Either (a) move prompts entirely into the slash command instruction file, OR (b) keep prompts in Python as constants that `build_translation_payload()` returns in the payload dict (so the slash command receives the pre-built prompt). Option (b) is better — keeps domain knowledge (how to translate legal documents) in the Python layer where it belongs logically.

**Warning signs:** Slash command has duplicated prompt text that also exists in Python files.

### Pitfall 2: Smoke Test Step 1 Becomes Meaningless

**What goes wrong:** `smoke_test.py` Step 1 (`step1_credential_check`) calls `load_api_key()`. After deleting `env_loader.py`, this step fails or becomes a no-op.

**Why it happens:** The smoke test was designed to validate API connectivity end-to-end. After refactor, there are no credentials to check.

**How to avoid:** Decide smoke test fate explicitly. Options:
- **Remove** `smoke_test.py` entirely (no longer validates anything meaningful)
- **Replace Step 1** with "verify structure detector imports correctly" (keeps the tool useful for code health checks)
- **Adapt**: Remove Step 1, keep Steps 2-5 (structure detection → formatting → pipeline integrity) as a pure Python sanity check

**Recommendation:** Keep smoke_test.py but replace Step 1 with a Python import/module-health check. Steps 2-5 remain valuable for testing the non-LLM parts of the pipeline.

### Pitfall 3: `verify.md` Calls `TranslationVerifier` Directly

**What goes wrong:** The `/verify --translation` command currently describes instantiating `TranslationVerifier()`, which requires an API key. After refactor, `TranslationVerifier.__init__()` will no longer need API setup, but the slash command instructions may still contain stale API-initialization language.

**Why it happens:** Slash commands reference Python class constructors. If the class signature changes, the command instructions become stale.

**How to avoid:** After refactoring `verification.py`, update `verify.md` to reflect the new (no-arg or simplified) constructor. The audit mode of `TranslationVerifier` is already pure Python (regex parsing) — only the fresh mode and scholarly comparison modes need LLM refactoring.

### Pitfall 4: Cross-Checker Has Two Types of External Calls

**What goes wrong:** `cross_checker.py` calls BOTH DeepL/googletrans (third-party translation backends) AND the Anthropic API (for semantic comparison). When removing only the Anthropic calls, the DeepL/googletrans logic must remain intact.

**Why it happens:** The module's two responsibilities (get independent translation + compare translations semantically) get conflated.

**How to avoid:** In `cross_checker.py`, only remove `_compare_translations()` (the `client.messages.create()` call). Keep `_translate_with_backend()` unchanged. The semantic comparison becomes a Claude Code operation in the slash command.

### Pitfall 5: env_loader.py is Imported in smoke_test.py

**What goes wrong:** `smoke_test.py` imports `from env_loader import load_api_key` at line 26. After deleting `env_loader.py`, the smoke test fails to import.

**Why it happens:** The smoke test was tightly coupled to credential checking.

**How to avoid:** Remove the `env_loader` import from `smoke_test.py` before deleting `env_loader.py`. The smoke test's Step 1 needs to be rewritten or removed first.

## Open Questions

1. **Should `cli.py` survive the refactor?**
   - What we know: `cli.py` currently provides `python tools/translation/cli.py translate ...` as a standalone invocation. After refactor, the LLM calls are in the slash command layer, so `cli.py`'s translate/crosscheck/annotate/surprise subcommands would be broken shells.
   - What's unclear: Whether any user or workflow actually calls `cli.py` directly (vs via slash commands). The `detect` and `smoketest` subcommands could survive.
   - Recommendation: Keep `cli.py` but gut the LLM-dependent subcommands (translate, crosscheck, annotate, surprise). Keep `detect` (pure Python) and `smoketest` (after smoke_test.py update). Mark removed commands with clear error messages.

2. **Cross-check semantic comparison model selection**
   - What we know: The semantic comparison task (are these two translations semantically equivalent?) is a structured JSON output task, relatively simple.
   - What's unclear: Whether `claude-haiku-4-5` is a valid current alias, or whether haiku generation naming has changed since training.
   - Recommendation: Default to `claude-sonnet-4-6` for all steps in the initial implementation. Optimize per-step model selection as a follow-up once the architecture is validated.

3. **Does `verification.py` audit mode need LLM at all?**
   - What we know: `_audit_existing_output()` is pure Python regex parsing — no LLM. Only `_compare_against_scholarly_summary()` and `_compare_against_claude_knowledge()` call the API.
   - What's unclear: How often the scholarly comparison is actually used in practice.
   - Recommendation: Audit mode needs zero changes. Fresh mode and scholarly comparison become Claude Code operations. The `TranslationVerifier` class simplifies significantly.
